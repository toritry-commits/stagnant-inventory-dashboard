## このダッシュボードでできること

このダッシュボードは、**減損が予定されている在庫**を一覧で確認できるツールです。
**毎朝9時に自動更新**され、常に最新のデータを確認できます。

### 主な機能

| 機能 | 説明 |
|------|------|
| **自動更新** | 毎朝9時にBigQueryからデータを取得して自動更新 |
| **表示モード切替** | 期末1のみ / 期末1・2のみ / 全期間 の3種類から選択 |
| **商品画像の自動表示** | A列に商品画像が自動で表示される |
| **手入力データの保持** | BP列以降に入力したデータは更新しても消えない |
| **削除履歴の自動保存** | 減損完了したSKUは「削除履歴」シートに自動保存 |

### 主な用途

| 目的 | 確認すべきポイント |
|------|-------------------|
| **減損回避** | 簿価が高い商品から優先的に対策を検討 |
| **滞留解消** | 在庫数が多い商品、長期滞留している商品を特定 |
| **期末準備** | 次の期末までに減損予定の在庫を事前把握 |

### 減損とは？

在庫が**365日以上**倉庫に滞留すると、会計上「減損」として処理されます。
減損処理は**期末（2月末/5月末/8月末/11月末）**に実施されます。

## 基本的な使い方

### スプレッドシートを開くだけ

このダッシュボードは以下が自動化されています：

- **除外対象は表示されない**: BigQueryで取得時に絞り込み済み
- **画像は自動表示**: A列にIMAGE関数が自動設定
- **ソート済み**: 期末1の減損予定簿価が大きい順に並び替え済み
- **表示モードはデフォルトで「期末1のみ」**: 必要な列だけ表示

**特別な操作なしで、すぐに分析を始められます。**

### シートの構成

| シート名 | 内容 |
|----------|------|
| **滞留在庫レポート** | メインのデータシート（毎日更新） |
| **削除履歴** | 減損完了で消えたSKUの履歴（自動保存） |

### 列の構成

| 列範囲 | 内容 |
|--------|------|
| A列 | 商品画像（自動表示） |
| B〜BO列 | BigQueryから取得したデータ |
| BP列以降 | 手入力データ（更新しても消えない） |

## メニューの使い方

スプレッドシート上部の **「滞留在庫レポート」** メニューから操作できます。

### メニュー構成

```
滞留在庫レポート
├── 期末1の列だけ表示      ← 次の期末のみ確認したいとき
├── 期末1・2の列だけ表示   ← 半年先まで確認したいとき
├── すべての列を表示       ← 全期間を確認したいとき
├── ─────────────────────
└── データを今すぐ更新     ← 最新データに手動更新
```

### 表示モードの違い

| モード | 表示される列 | 用途 |
|--------|-------------|------|
| **期末1の列だけ表示** | 次の期末（約3ヶ月後）に関する列のみ | 直近の減損対策に集中したいとき |
| **期末1・2の列だけ表示** | 2期先（約6ヶ月後）までの列 | 半年先まで見通したいとき |
| **すべての列を表示** | 全期間（期末1〜4 + 減損済）の列 | 全体像を把握したいとき |

### データの更新について

- **自動更新**: 毎朝9時に自動で最新データに更新されます
- **手動更新**: 「データを今すぐ更新」で即座に最新化できます
- **処理時間**: 約1〜3分（データ量による）

## 目的別の活用方法

### 目的1: 減損回避（金額インパクトを減らしたい）

**見るべき列**: AY列「期末1_減損予定_簿価」

**手順**:
1. スプレッドシートを開く（デフォルトで期末1の簿価が大きい順）
2. 上位の商品から減損回避策を検討

**ポイント**:
- 簿価が高い商品 = 減損時の金額インパクトが大きい
- デフォルトで簿価の大きい順に並んでいるので、上から順に対策を検討

---

### 目的2: 滞留解消（在庫を動かしたい）

**見るべき列**: AC列「期末1_商品数」、AI列「期末1_余りパーツ数」

**手順**:
1. スプレッドシートを開く
2. 商品数や余りパーツ数が多い商品を確認
3. 滞留解消策を検討（販促、セット販売、処分など）

**ポイント**:
- 「商品数」= 完成品として組み上げられる在庫
- 「余りパーツ数」= 商品として組み上げられない余りの部品

---

### 目的3: 期末ごとの影響を把握したい

**手順**:
1. メニューから「すべての列を表示」を選択
2. 期末1〜4の各列を比較

**ポイント**:
- 「期末1_日付」〜「期末4_日付」列で、具体的な期末日を確認できます
- 例: 今日が1月なら、期末1 = 2月末、期末2 = 5月末...

---

### 目的4: 特定の商品を詳しく調べたい

**見るべき列**: 構成パーツid、構成パーツ名、在庫id

**手順**:
1. 調べたい商品のSKU_IDまたは商品名でフィルター
2. 「構成パーツid」「構成パーツ名」で、何のパーツで構成されているか確認
3. 「在庫id」列で、具体的にどの在庫が対象か確認

**在庫IDの見方**:
```
[パーツid:倉庫名]在庫id, 在庫id, ...

例: [8049:船橋]123456, 123457
    ↑パーツ8049が船橋倉庫にあり、在庫ID 123456と123457が対象
```

---

### 目的5: 手入力でメモを残したい

**BP列以降**に自由にデータを入力できます。

**特徴**:
- データ更新しても消えない（SKU IDで紐づけて自動復元）
- 対応状況、担当者、コメントなどを記録可能

**注意**: ヘッダー（1行目）は自分で入力してください。

## 各項目の説明

### 商品情報（B〜O列付近）

| 列名 | 説明 |
|------|------|
| 画像リンク | 商品画像のURL（A列に自動表示） |
| 除外フラグ | 通常は空欄（「除外」のデータはBigQueryで除外済み） |
| SKU_ID | 商品のSKU識別番号 |
| 商品ID | 商品の識別番号（管理画面で使用） |
| 商品IDリンク | 管理画面の商品ページへのリンク |
| シリーズ名 | 商品シリーズの名前 |
| 商品名 | 商品の名前 |
| 属性 | 色やサイズなどの属性情報 |
| カテゴリ | ソファ、ベッド、テーブルなどの分類 |
| 対象顧客 | 一般顧客向けか法人顧客向けか |
| サプライヤー | 仕入先 |
| 下代 | 仕入れ価格 |
| 上代 | 販売価格 |

### 構成情報（P〜R列付近）

| 列名 | 説明 |
|------|------|
| 構成パーツid | 商品を構成するパーツのID一覧 |
| 構成パーツ名 | パーツの名前一覧 |
| 構成点数 | 1商品に必要なパーツの数量 |

### 在庫ID（S〜AB列付近）

| 列名 | 説明 |
|------|------|
| 期末1_減損予定_在庫id_商品 | 期末1までに減損予定の商品用在庫ID |
| 期末1_減損予定_在庫id_余りパーツ | 期末1までに減損予定の余りパーツ在庫ID |
| （期末2〜4、減損済も同様） | |

### 数量（AC〜AT列付近）

| 列名 | 説明 |
|------|------|
| 期末1_減損予定_商品数 | 期末1までに減損予定の完成品数 |
| 期末1_減損予定_余りパーツ数 | 期末1までに減損予定の余りパーツ数 |
| 期末1_減損予定_商品+余りパーツ | 上記の合計 |
| （期末2〜4、減損済も同様） | |

### 累計（AU〜AX列付近）

| 列名 | 説明 |
|------|------|
| 期末1_減損予定_商品+余りパーツ累計 | 期末1までの累計 |
| 期末2_減損予定_商品+余りパーツ累計 | 期末1 + 期末2の累計 |
| 期末3_減損予定_商品+余りパーツ累計 | 期末1〜3の累計 |
| 期末4以降_減損予定_商品+余りパーツ累計 | 期末1〜4以降の累計（全体） |

### 簿価（AY〜BF列付近）

| 列名 | 説明 |
|------|------|
| 期末1_減損予定_簿価 | 期末1で減損予定の在庫の簿価（**ソート基準**） |
| 期末1_減損予定_簿価_累計 | 期末1までの簿価累計 |
| （期末2〜4も同様） | |

### 平均簿価（BG〜BJ列付近）

| 列名 | 説明 |
|------|------|
| 期末1_減損予定_平均簿価 | 期末1で減損予定の在庫1点あたりの平均簿価 |
| （期末2〜4も同様） | |

### その他（BK〜BO列付近）

| 列名 | 説明 |
|------|------|
| 滞留日数_平均_切上 | そのSKUの在庫が平均何日滞留しているか |
| 期末1_日付 | 次の期末日（例: 2025-02-28） |
| 期末2_日付〜期末4_日付 | 2番目〜4番目の期末日 |

## よくある質問

### Q1: データはいつ更新される？

**毎朝9時に自動更新**されます。
すぐに最新データが必要な場合は、メニューから「データを今すぐ更新」を実行してください。

### Q2: 「商品数」と「余りパーツ数」の違いは？

**商品数**: 完成品として組み上げられる在庫の数です。
例えば、ベッドフレームが「本体3点 + 脚1点」で構成される場合、本体6点と脚2点があれば「2商品」組み上げられます。

**余りパーツ数**: 商品として組み上げられない余りのパーツです。
上記の例では、本体が6点で脚が2点の場合、脚が足りないため本体4点が余ります。

### Q3: 期末1、期末2... はいつのこと？

実行日によって自動的に決まります。
- **期末1**: 次に来る期末日（2月末/5月末/8月末/11月末のうち最も近い日）
- **期末2**: その次の期末日
- **期末3**: 3番目に近い期末日
- **期末4以降**: 4番目以降の期末日

具体的な日付は「期末1_日付」〜「期末4_日付」列で確認できます。

### Q4: 「減損済」とは？

すでに365日以上滞留して減損処理済みの在庫です。
今後の減損対策では「期末1〜4」の数値を見てください。

### Q5: 手入力したデータは更新で消える？

**消えません。** BP列以降に入力したデータは、SKU IDで紐づけて自動的に復元されます。
対応状況やメモなどを自由に記録してください。

### Q6: 削除されたSKUはどこで確認できる？

「**削除履歴**」シートに自動保存されています。
減損完了などで一覧から消えたSKUの履歴を確認できます。

### Q7: 列が多すぎて見づらい

メニューから表示モードを切り替えてください。
- **期末1の列だけ表示**: 直近の対策に集中したいとき
- **期末1・2の列だけ表示**: 半年先まで確認したいとき

### Q8: 「除外フラグ」でフィルターする必要がある？

**いいえ、不要です。** BigQueryで取得時に除外対象は除外済みです。
表示されているデータはすべて分析対象です。

## 注意事項

### 縦合計は正確ではありません

このデータは**SKU単位**で集計されています。
同じ在庫が複数のSKUで使用される場合があるため、**縦方向の合計（SUM）は二重カウントになる可能性があります**。

```
例: パーツAが SKU_ID 10 と SKU_ID 15 の両方に使われている場合、
    両方の行にパーツAの数量・簿価が含まれます。
    → 縦合計すると二重カウントになります
```

**正確な合計が必要な場合**は、別途パーツ単位での集計をご依頼ください。

### データは実行時点のスナップショット

このデータはクエリを実行した時点の在庫状況です。
日々の入出荷により数値は変動しますので、最新状況を確認したい場合は「データを今すぐ更新」を実行してください。

### ヘッダー行（1行目）は編集しないでください

1行目のヘッダーは更新処理で使用します。
変更すると正しく動作しなくなる可能性があります。
